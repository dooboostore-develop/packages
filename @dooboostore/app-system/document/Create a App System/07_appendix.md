# 부록: 더 나아가기

이 책을 통해 우리는 `@dooboostore/app-system`이라는 공통 애플리케이션 시스템의 설계와 구현 과정을 심층적으로 탐구했습니다. `ApiService`를 통한 HTTP 통신, `Alert` 시스템을 통한 사용자 피드백, 재사용 가능한 UI 컴포넌트, 상태 관리 및 프록시 시스템, 그리고 `Simple-Boot` 생태계와의 연동까지, 다양한 기능들을 어떻게 모듈화하고 통합하는지 살펴보았습니다. 이 부록에서는 `app-system` 아키텍처의 장단점을 객관적으로 평가하고, 시스템의 발전 방향에 대한 아이디어를 공유하겠습니다.

## A. Simple-Boot-App-System 아키텍처의 장단점

모든 소프트웨어 아키텍처에는 고유한 장점과 단점이 있습니다. `app-system`도 예외는 아닙니다.

### 장점

-   **모듈화된 공통 기능:** 애플리케이션 개발에 필요한 공통적이고 횡단적인 기능들을 독립적인 모듈로 제공하여 코드 중복을 줄이고 재사용성을 높입니다.
-   **DI 기반:** `simple-boot-core`의 DI 컨테이너를 활용하여 객체 간의 의존성을 느슨하게 만들고 테스트 용이성을 향상시킵니다.
-   **확장 가능한 `ApiService`:** 계층적 구조와 인터셉터 패턴을 통해 HTTP 통신 로직을 유연하게 확장하고 커스텀 로직을 삽입할 수 있습니다.
-   **반응형 상태 관리:** `rxjs`를 활용한 `Store`는 상태 변화를 효율적으로 전파하여 여러 컴포넌트나 서비스가 동일한 상태를 공유하고 반응할 수 있도록 합니다.
-   **재사용 가능한 UI 컴포넌트:** `dom-render` 기반의 UI 컴포넌트들은 선언적이고 반응적이며, 개발 생산성을 높입니다.
-   **프레임워크 연동:** `Simple-Boot` 생태계의 다른 프레임워크들과 긴밀하게 연동되어 일관된 개발 경험을 제공합니다.

### 단점

-   **생태계 의존성:** `simple-boot-core`, `simple-boot-front`, `dom-render` 등 특정 `@dooboostore` 프레임워크들에 강하게 의존하므로, 다른 프레임워크와 함께 사용하기 어렵습니다.
-   **학습 곡선:** `simple-boot` 생태계의 여러 프레임워크 개념을 이해해야 하므로, 처음 접하는 개발자에게는 학습 곡선이 존재할 수 있습니다.
-   **디버깅의 복잡성:** `Proxy` 기반의 `ApiService` 인터셉터나 `dom-render`의 반응성 시스템은 코드의 실행 흐름을 가로채기 때문에, 예상치 못한 동작이 발생했을 때 디버깅이 복잡할 수 있습니다.

## B. 시스템의 발전 방향

`app-system`을 한 단계 더 발전시키기 위한 몇 가지 아이디어입니다.

-   **플러그인 시스템 강화:** `AlertFactory`나 `ApiServiceInterceptor`와 같은 확장 지점을 더욱 명확히 하고, 외부 개발자가 쉽게 플러그인을 개발하여 시스템에 기여할 수 있도록 문서화 및 가이드를 강화할 수 있습니다.
-   **테스트 커버리지 확대:** 각 모듈과 기능에 대한 단위 테스트 및 통합 테스트 커버리지를 확대하여 시스템의 안정성과 신뢰성을 높일 수 있습니다.
-   **성능 최적화:** `ApiService`의 네트워크 요청 최적화(예: HTTP/2 푸시, 웹 워커를 통한 백그라운드 요청), `Store`의 상태 업데이트 최적화 등을 통해 전반적인 성능을 향상시킬 수 있습니다.
-   **국제화(i18n) 지원:** 다국어 지원을 위한 통합된 메커니즘을 `Alert` 메시지나 UI 컴포넌트 텍스트에 적용할 수 있습니다.
-   **접근성(Accessibility) 강화:** 내장 UI 컴포넌트들이 웹 접근성 표준을 준수하도록 개선하여, 더 많은 사용자가 애플리케이션을 쉽게 사용할 수 있도록 할 수 있습니다.

이 책이 여러분의 공통 애플리케이션 시스템 개발 여정에 든든한 길잡이가 되었기를 바랍니다. 끊임없이 배우고, 탐구하며, 더 나은 소프트웨어를 만들어가는 즐거움을 만끽하시길 응원합니다.
